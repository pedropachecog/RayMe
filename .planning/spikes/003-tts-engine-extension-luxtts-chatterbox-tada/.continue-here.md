<current_state>
Spike 003 has moved past the original narrow TTFA test and the later methodology blocker. The shared chunk planner is now implemented in `.planning/phases/00-measurement-gate/probes/tts_scenario_matrix.py`, and the full Windows plus WSL matrix has been rerun on OMEN-PC.

The matrix covers native Windows and WSL, baseline and optimized profiles, and emits listenable WAV samples. Optimized rows now use `shared_chunked_playback` or `shared_chunked_streaming` where applicable.

XTTS long-form is no longer invalid: optimized long-form uses three planner chunks below the conservative token cap, `mode=shared_chunked_streaming`, `true_streaming=true`, and no streaming fallback.

The spike is closed as PASS_WITH_CAVEATS. Manual listening is still incomplete, but enough risk evidence exists to move on without making a bad default decision. User listening on 2026-04-23 has now scored the key Chatterbox long samples, all LuxTTS optimized samples, TADA optimized long samples, WSL XTTS optimized long, and Windows F5 optimized long:
- Chatterbox Turbo baseline long-form is gibberish on both Windows and WSL.
- Chatterbox Turbo normal optimized long-form is fine on both Windows and WSL.
- Chatterbox Turbo `optimized_seed_1337` long-form is fine on both Windows and WSL.
- LuxTTS optimized short/medium/long on both Windows and WSL has sample-level quality failures: reference/sample tail repetition, metallic noise, over-fast delivery, sample phrase leakage, poor speaker match, and accent loss.
- TADA 1B Windows optimized long sounds acceptable with some hiccups; WSL optimized long is worse, less steady, morphs toward another voice, and has hiccups.
- WSL XTTS optimized long sounds metallic and hiccupy with incomplete voice likeness; XTTS remains sample-sensitive based on prior user testing.
- Windows F5 optimized long sounds stretched/slow in parts, hiccupy, and can degrade into mumbling/silence before recovering; the 1.5 stretch/duration multiplier is likely too much.

Do not interpret those LuxTTS rows as an engine-removal decision. The user explicitly asked to keep LuxTTS and every other engine in future implementation; the quality notes should guide defaults, labels, warnings, and retesting priorities.
</current_state>
