<current_state>
Spike 003 has moved past the original narrow TTFA test. A new warm-model scenario matrix now measures RayMe-style TTS requests after model load and warmup are complete. The matrix covers native Windows and WSL, baseline and optimized profiles, and emits listenable WAV samples.

The numeric result favors LuxTTS for warm total latency, but the spike remains PARTIAL because manual listening exposed quality failures. The user specifically reported that Chatterbox Turbo long samples are pure gibberish and that some other samples are subjectively unacceptable.

Important methodology correction: XTTS long-form streaming was not properly measured. The harness hit XTTS's 400-token `inference_stream` cap and fell back to full-render timing instead of segmenting the long text. Those rows should be treated as invalid for XTTS streaming decisions until token-aware splitting is implemented and the matrix is rerun.

Broader methodology correction: chunking must be implemented for all engines, not just XTTS. Engines without native streaming need a best practical chunked playback path so their benchmark reflects how RayMe would actually speak a long LLM answer.
</current_state>

<blockers>
- Manual quality is not yet scored in a structured artifact.
- XTTS long-form streaming requires a harness fix: split long text before `inference_stream` so no segment reaches the 400-token cap, then measure first-segment TTFA and total stitched playback time.
- All engines need a shared chunk planner for long-form testing. It should prefer sentence boundaries, enforce engine-specific token/character caps, avoid tiny fragments, measure first-chunk TTFA, total stitched time, inter-chunk gaps, and emit a stitched WAV for listening.
- Chatterbox Turbo long-form output failed subjective listening.
- TADA 1B native Windows path failed with a CUDA error during warmup/generation.
- The harness uses fixed canned scenarios rather than a corpus of real RayMe LLM answers.
</blockers>
