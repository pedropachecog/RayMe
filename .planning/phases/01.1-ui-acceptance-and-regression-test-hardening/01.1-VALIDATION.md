# Phase 01.1 - Validation Strategy

Per-phase validation contract for feedback sampling during execution.

## Test Infrastructure

| Property | Value |
|----------|-------|
| Framework | Playwright 1.59.1 for browser E2E; Vitest 4.1.5 for client unit; pytest 9.0.3 for backend/API |
| Config file | `web-ui/client/playwright.config.ts`, `web-ui/client/vitest.config.ts`, `web-ui/server/pyproject.toml` |
| Quick run command | `npm --prefix web-ui/client run test:e2e -- portrait-persistence.spec.ts settings-connection.spec.ts` |
| Full suite command | `uv run --project web-ui/server pytest web-ui/server/tests -q && npm --prefix web-ui/client run test:unit -- --run && npm --prefix web-ui/client run test:e2e` |
| Estimated runtime | Unknown until the new Playwright specs exist; executor must record observed runtime in the phase summary |

## Sampling Rate

- After every task commit: run the narrow E2E/API command for the touched surface.
- After every client-code task commit: also run `npm --prefix web-ui/client run test:unit -- --run`.
- After every plan wave: run `uv run --project web-ui/server pytest web-ui/server/tests -q && npm --prefix web-ui/client run test:unit -- --run && npm --prefix web-ui/client run test:e2e`.
- Before `$gsd-verify-work`: full local suite must be green, live `OMEN-PC` browser smoke must be green with console/page-error capture, then product-owner Android Chrome checkpoint may be requested.
- Max feedback latency: no three consecutive implementation tasks may land without an automated verification command.
