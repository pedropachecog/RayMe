- Any visual correction made for acceptance must be minimal and must remain compatible with the approved Phase 1 UI-SPEC.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, dense message action spacing |
| sm | 8px | Compact element spacing, message stack gaps, grid gutters on mobile |
| md | 16px | Default element spacing, input padding, card internal padding |
| lg | 24px | Section padding, editor form group spacing, desktop grid gutters |
| xl | 32px | Page gutters, shell content gap, chat rail-to-thread gap |
| 2xl | 48px | Major vertical separation, empty-state breathing room |
| 3xl | 64px | Desktop top/bottom workspace padding only |

Exceptions: minimum interactive target is 44px for icon buttons, nav items, card action menus, portrait replace/remove controls, message overflow controls, Settings `Test Connection` buttons, and mobile composer actions.

Non-regression requirements:

- Portrait fallback, loaded portrait, replacement, and removed states must keep the same card, row, header, and picker dimensions. No image load or missing image state may resize a Gallery card, Home thread row, Home character picker row, Chat header, or Editor preview container.
- Browser error/status chips added for acceptance must fit within existing Settings and shell panels without changing page-level gutters.
- Mobile checkpoint layouts must keep composer, bottom nav, dialogs, and Settings actions non-overlapping at the existing Pixel 5 Playwright viewport.

| Label | 12px | 600 | 1.3 |

Typography rules:

- Use only weights 400 and 600.
- Letter spacing is 0.
- Acceptance-visible status labels use Label, never Display.
- Error and handoff-related copy inside the app must fit within existing panels at desktop and mobile widths.
- Long URLs and model names, including `http://192.168.1.190:8001/v1` and `unsloth/Qwen3.5-27B`, must wrap or truncate without forcing horizontal scroll outside text inputs.

## Acceptance Surface Contracts

### Browser Observability

- Every new or modified Phase 01.1 Playwright spec must install a shared browser error guard that captures `page.on('console')` error messages and `page.on('pageerror')` exceptions.
- A test may allowlist a known benign console error only with a narrow matcher and an inline reason. Broad console-error suppression is not allowed.
- Browser readiness requires rendered content assertions. HTTP 200, `/health`, or `curl` output alone does not satisfy UI readiness.
- Blank page, missing app bootstrap, CSP bootstrap failure, broken image request, and uncaught page exception are acceptance failures.

### Portrait Persistence

- PNG card import must prove the persisted portrait renders through an actual browser image request, not only through a `portrait_url` JSON assertion.
- Direct portrait upload must prove preview, save, reload, and rendered image persistence.
- Required surfaces after import/upload: Character Editor portrait preview, Character Gallery card, Home start-chat picker, Home thread list row, Chat header, and post-reload state.
- Replacement must update every required surface to the new image URL or asset identity.
- Removal must update every required surface to the existing initials/fallback avatar after save/reload, with no broken image icon and no stale portrait.
- Tests must include deletion/replacement behavior for both browser-visible DOM and relevant API persistence where the backend owns the state.

### Settings Connection Tests

- Pressing any Settings `Test Connection` button must first persist the current form values through Settings save/PATCH behavior, then invoke the corresponding test endpoint.
- The Web UI, AI backend, and LLM sections each keep their own visible status: `Connected`, `Unreachable`, `Unauthorized`, or `Not configured`.
- The local OpenAI-compatible LLM acceptance path uses `http://192.168.1.190:8001/v1` and `unsloth/Qwen3.5-27B` as Settings-entered test data only. Production code must not hardcode either value.
- A blank LLM API key is valid for the local no-key server path. The UI must not force a key before running that local connection test.
- API key values remain masked and must not appear in status copy, toast copy, console output, or screenshots intended for handoff.

### Full Phase 1 Text Path

- The browser path must cover import or create, edit character, save, start chat, alternate greeting selection, streaming reply, regenerate, swipe/generate alternate, continue with composer text, reload, and continue again.
- Streaming AI output remains a single message bubble with token-by-token text and the approved accent caret. Completion must replace the streaming buffer without a layout jump.
- `Redo and Replace` replaces the active AI turn, not appends a second canonical answer.
- Swipe state shows only the selected alternate as canonical for continuation.
- Continue uses composer text when present.
- Stale downstream turns retain the approved subtle rail and `Stale` chip behavior.
- The phase must preserve existing mobile text-flow coverage and chat virtualization coverage.

### Live LAN And Android Checkpoint

- Local deterministic Playwright and API checks come first.
- Live `OMEN-PC` verification comes second for behavior depending on LAN, HTTPS, persisted Settings, or deployed runtime. Browser smoke must use `https://192.168.1.199:8443`; AI backend health is `https://192.168.1.199:9443/health`.
- Physical Android Chrome product-owner acceptance comes last. The agent handoff must explicitly list the passed automated checks and ask only for the remaining phone-specific signal.
- Do not create `01-24-SUMMARY.md` or mark Phase 1 acceptance complete until hardened local suite, live LAN browser smoke, and Android Chrome checkpoint pass.

## Interaction And State Contracts

### Loading

- Use existing tonal skeletons or compact inline progress for import, save, connection test, streaming, and reload states.
- Connection-test loading state must remain scoped to the endpoint being tested.
- Portrait image loading must reserve stable dimensions and then reveal the image without layout shift.

### Empty

- Empty Gallery and empty Home states keep Phase 1 copy and actions.
- Removed portrait is not an empty state. It is the normal initials/fallback avatar state.
- No empty Chat state should appear after starting a chat because the opening greeting is inserted immediately.

### Error

- Settings errors identify the endpoint that failed and show the existing status vocabulary.
- Portrait errors distinguish unsupported image, failed upload, failed removal, and failed image load when surfaced to the user.
- Import errors distinguish unsupported file, malformed JSON, unreadable PNG metadata, and unsafe content.
- LLM streaming failure keeps the user message and shows the approved retry/regenerate path.

### Focus And Keyboard

- Existing keyboard focus behavior must remain visible after adding selectors or acceptance hooks.
- Icon-only buttons for portrait replace/remove, message actions, API key reveal, and mobile navigation require accessible labels.
- Dialogs used in the hardened paths must trap focus and close on Escape unless a destructive action is submitting.

### Motion

- Preserve existing reduced-motion handling in `app.css`.
- Do not add new decorative animation for this phase.
- Streaming caret, skeleton shimmer, and status transitions remain the only acceptable motion changes.

## Responsive Contract

- All hardened paths must pass both Playwright projects: `desktop-chromium` and `mobile-chromium`.
- Mobile Settings, portrait upload, import dialog, alternate greeting dialog, chat composer, and bottom nav must not overlap.
- Thread titles, character names, URLs, model names, and status labels must not overflow buttons, cards, rows, chips, dialogs, or the mobile viewport.
- The Android Chrome manual checkpoint must use the live LAN URL and existing trusted mkcert flow; do not ask for a new certificate install unless the reusable root CA has changed.

## Agent-To-User Handoff Contract

When asking for manual Android Chrome acceptance, use this structure:

1. State the completed agent checks: backend/API command(s), Playwright browser command(s) with console/page-error capture, and live LAN browser smoke result.
2. State the exact phone URL: `https://192.168.1.199:8443`.
3. Ask for the narrow product-owner signal: secure page load, Settings connection states, portrait visibility after reload across Gallery/Home/Chat, and full text flow usability.
4. Do not ask the user to debug, inspect logs, or discover first-line regressions that the agent can test.

Required handoff copy stem:

```text
Agent verification completed: backend/API checks, Playwright browser checks with console and page-error capture, and live LAN browser verification have passed. Please confirm the remaining Android Chrome product-owner signal at https://192.168.1.199:8443.
```
