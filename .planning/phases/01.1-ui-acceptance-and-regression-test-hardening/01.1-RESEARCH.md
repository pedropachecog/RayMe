# Phase 01.1 Context: UI Acceptance and Regression Test Hardening

## Why This Phase Exists

Phase 1 plan 01-24 reached live LAN/Android acceptance before enough of the UI
surface had been exercised by the agent. The user found workflows that appeared
implemented but were not actually working end to end:

- Svelte app initially rendered as a blank page because CSP blocked the bootstrap.
- Imported PNG character cards created characters without a persisted portrait.
- Direct portrait upload returned success but the image disappeared because the
  API did not expose or serve `portrait_url`.
- Settings "Test Connection" tested stale saved settings, so a typed local LLM URL
  still probed the default OpenAI endpoint and returned `Unauthorized`.

This phase exists to convert those discoveries into durable automated coverage
and to make the acceptance workflow respectful of the product owner role.

## Non-Negotiable Operating Rule

The user's manual testing is product-owner acceptance, not first-line QA. Before
asking for manual Android/Chrome testing, the agent must already have run the
relevant checks itself:

- Backend/API tests for the behavior being claimed.
- Browser-level Playwright checks for rendered UI and console/page errors.
- Live deployed verification on `OMEN-PC` when the behavior depends on LAN,
  HTTPS, persisted settings, or the real backend runtime.

When asking for product-owner testing, state what has already been verified and
what acceptance signal is still needed.

### Pattern 1: Shared Playwright Error Guard

**What:** Attach `page.on('console')` and `page.on('pageerror')` in a helper or fixture and fail tests on unexpected browser errors. [CITED: https://github.com/microsoft/playwright.dev/blob/main/nodejs/versioned_docs/version-stable/api/class-consolemessage.mdx; CITED: https://github.com/microsoft/playwright.dev/blob/main/nodejs/versioned_docs/version-stable/api/class-page.mdx]

**When to use:** Use in every Phase 01.1 E2E spec, because CONTEXT.md requires console and page-error capture. [VERIFIED: CONTEXT.md]

**Example:**

```typescript
// Source: Playwright console/pageerror docs + local Playwright test style.
import { expect, type Page } from '@playwright/test';

export function installBrowserErrorGuard(page: Page) {
  const consoleErrors: string[] = [];
  const pageErrors: string[] = [];

  page.on('console', (message) => {
    if (message.type() === 'error') {
      consoleErrors.push(message.text());
    }
  });

  page.on('pageerror', (error) => {
    pageErrors.push(String(error));
  });

  return async function expectNoBrowserErrors() {
    expect(pageErrors, 'uncaught page errors').toEqual([]);
    expect(consoleErrors, 'browser console errors').toEqual([]);
  };
}
```

## Common Pitfalls

### Pitfall 1: Route Mocks That Do Not Prove Images Render

**What goes wrong:** Tests assert `portrait_url` data exists but never serve the image or assert a browser-visible `<img>`. [VERIFIED: existing E2E fixtures use `portrait_url: null`]
**Why it happens:** API response assertions are easier than UI surface traversal. [ASSUMED]
**How to avoid:** Fulfill the portrait URL with image bytes and assert visible `<img>` across Editor, Gallery, Home, Chat, reload, replacement, and removal. [VERIFIED: CONTEXT.md]
**Warning signs:** E2E specs contain `portrait_url: null` or only check JSON payloads. [VERIFIED: web-ui/client/tests/e2e]

### Pitfall 2: Settings Test Uses Persisted Defaults Instead Of Typed Form Values

**What goes wrong:** The user types a local LLM URL, but the test endpoint probes the old saved OpenAI default. [VERIFIED: CONTEXT.md]
**Why it happens:** Backend test routes read server-side Settings, not POST body overrides. [VERIFIED: web-ui/server/app/api/settings.py; VERIFIED: web-ui/server/tests/test_health_settings.py]
**How to avoid:** E2E must assert `PATCH /api/settings` completes with current form values before each `POST /api/settings/test/*`. [VERIFIED: web-ui/client/src/routes/settings/+page.svelte]
**Warning signs:** A Settings E2E expects `POST /api/settings/test/llm` without a preceding PATCH assertion. [VERIFIED: tests/unit/settings.test.ts currently unit-tests wrappers but not full UI order]

### Pitfall 3: Local Playwright Passes But Live LAN Runtime Is Old

**What goes wrong:** Local tests pass against current source, while `OMEN-PC` still serves an older checkout or copied staging tree. [VERIFIED: OPERATING-NOTES.md]
**Why it happens:** Plan 01-24 had a temporary copied staging tree and the durable checkout must be verified before services start. [VERIFIED: OPERATING-NOTES.md]
**How to avoid:** Plan a live gate that records `git status`, branch, and commit from `C:\Users\pmpg\rayme\RayMe\`, then browser-checks `https://192.168.1.199:8443`. [VERIFIED: web-ui/server/docs/HTTPS-LAN.md]
**Warning signs:** The agent relies on curl only, does not capture rendered UI, or cannot resolve the SSH alias. [VERIFIED: curl probe; VERIFIED: ssh probe; VERIFIED: OPERATING-NOTES.md]

### Pitfall 4: Product-Owner Test Request Omits Agent Evidence

**What goes wrong:** The user is asked to find defects the agent could have caught. [VERIFIED: CONTEXT.md]
**Why it happens:** Manual acceptance gets treated as QA. [VERIFIED: CONTEXT.md]
**How to avoid:** Any manual request must say which backend/API/browser/live checks passed and what product-owner signal remains. [VERIFIED: CONTEXT.md; VERIFIED: OPERATING-NOTES.md]
**Warning signs:** "Can you try it on your phone?" appears without a preceding checklist. [VERIFIED: CONTEXT.md]
