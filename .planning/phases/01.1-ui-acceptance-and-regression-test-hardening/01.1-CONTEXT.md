# Phase 01.1 Context: UI Acceptance and Regression Test Hardening

## Why This Phase Exists

Phase 1 plan 01-24 reached live LAN/Android acceptance before enough of the UI
surface had been exercised by the agent. The user found workflows that appeared
implemented but were not actually working end to end:

- Svelte app initially rendered as a blank page because CSP blocked the bootstrap.
- Imported PNG character cards created characters without a persisted portrait.
- Direct portrait upload returned success but the image disappeared because the
  API did not expose or serve `portrait_url`.
- Settings "Test Connection" tested stale saved settings, so a typed local LLM URL
  still probed the default OpenAI endpoint and returned `Unauthorized`.

This phase exists to convert those discoveries into durable automated coverage
and to make the acceptance workflow respectful of the product owner role.

## Non-Negotiable Operating Rule

The user's manual testing is product-owner acceptance, not first-line QA. Before
asking for manual Android/Chrome testing, the agent must already have run the
relevant checks itself:

- Backend/API tests for the behavior being claimed.
- Browser-level Playwright checks for rendered UI and console/page errors.
- Live deployed verification on `OMEN-PC` when the behavior depends on LAN,
  HTTPS, persisted settings, or the real backend runtime.

When asking for product-owner testing, state what has already been verified and
what acceptance signal is still needed.

## Required Test Coverage

Plan this phase around browser-visible workflows, not just API unit tests:

1. PNG card import persists portrait and shows it after reload in Editor, Gallery,
   Home character picker, and Chat.
2. Direct portrait upload persists portrait and shows it after reload.
3. Portrait replacement and removal update every surface predictably.
4. Settings connection tests save current form values before testing Web UI,
   AI backend, and LLM.
5. LLM Settings supports an OpenAI-compatible local server with no API key.
6. Full Phase 1 text flow works in browser: import/create/edit character, start
   chat, alternate greeting, stream reply, regenerate, swipe, continue, reload,
   and continue again.
7. All browser checks include console and page-error capture.
