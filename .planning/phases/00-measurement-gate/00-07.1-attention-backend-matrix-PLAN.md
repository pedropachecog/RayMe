<objective>
Benchmark optimization backends across the Phase 0 TTS engines and label the actual backend used in every persisted result.

Purpose: the current Qwen3 numbers on `OMEN-PC` are a no-FlashAttention eager-attention baseline, but the repo does not yet distinguish `eager`, `sdpa`, `flash_attention_2`, or `not_applicable` in a systematic way across engines. This plan closes that gap before Phase 0 writeback freezes the TTS decision.

Output: `results/tts_attention_matrix.json` plus updated TTS probe output that records the attention/acceleration backend actually used for each engine measurement.
</objective>

<context>
- Execution host for backend measurements remains `OMEN-PC` (`192.168.1.199`) over SSH.
- `00-07-fa2-install-PLAN.md` already established whether `flash-attn` can be installed on `OMEN-PC`.
- `00-08-synthesis-writeback-PLAN.md` must consume backend-labeled TTS results instead of ambiguous “fast”/“slow” numbers.
</context>

<must_haves>
- For each engine, record supported optimization modes and the mode actually used in the measurement.
- `Qwen3` must distinguish at minimum `eager` baseline vs `flash_attention_2` when FA2 is available; if FA2 is unavailable, record that explicitly.
- `F5` and `XTTS` must be marked with either a measured backend or an explicit `not_applicable` / `not_supported` label rather than silence.
- The resulting JSON must make it impossible to mistake a no-FA2 baseline for an optimized run.
</must_haves>
