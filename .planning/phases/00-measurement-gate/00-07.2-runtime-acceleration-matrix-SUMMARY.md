## Decisions Made

- Native Windows remained the fastest measured F5 path for both short-ack (`521.8 ms`) and long-form (`528.4 ms`) runs.
- WSL Python remained close to the native F5 baseline, while WSL Triton underperformed badly on short-ack and failed outright on long-form with an HTTP 500.
- XTTS DeepSpeed could not be used in the pinned WSL stack because of a mixed CPU/CUDA tensor device error, so the baseline WSL Python path stays recommended.
- Qwen FlashAttention 2 was runnable only in the WSL accelerator environment, but it was slower than the eager baseline (`4697.2 ms` vs `4514.0 ms` TTFA), so eager remains the recommended backend.

## Scope Boundary

- This plan compared runtimes and acceleration paths. It did not change the existing native Windows backend labeling artifact from `00-07.1`.
- The matrix is still OMEN-PC specific; it does not imply parity on other GPUs or on future CUDA/driver stacks.

## Issues Encountered

- F5 long-form on WSL Triton failed with a server-side HTTP 500, so that row is explicitly recorded as `not_available`.
- XTTS DeepSpeed initialization hit a CUDA-vs-CPU tensor placement error, so the accelerated row is recorded as unavailable rather than omitted.
- Qwen FlashAttention 2 did run in WSL, but it did not materially improve latency or throughput over eager on this hardware/software stack.
