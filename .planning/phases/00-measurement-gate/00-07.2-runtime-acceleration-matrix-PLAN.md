<objective>
Benchmark the omitted TTS runtime and acceleration permutations before final Phase 0 writeback freezes the stack.

Purpose: `00-07.1` made the current OMEN-PC backend labels explicit, but it did not answer the broader comparison the user asked for: native Windows vs WSL Python vs WSL Triton for F5, XTTS baseline vs DeepSpeed-enabled execution, and Qwen eager vs FlashAttention 2 when WSL makes the accelerated path available. This plan closes that gap so `00-08` can distinguish "measured on the native Windows probe" from "measured across the full candidate runtime matrix."

Output: `results/tts_runtime_matrix.json` plus any supporting raw artifacts needed to make the comparison reproducible.
</objective>

<context>
- Native Windows backend work remains on `OMEN-PC` via `ssh rayme-ssh`.
- WSL-backed accelerator work remains on `OMEN-PC` via `ssh rayme-pmpg` followed by direct `wsl -d Ubuntu ...` commands.
- `00-07.1` already labels the current native-Windows benchmark results and must be treated as the baseline, not as the full runtime matrix.
- `.planning/spikes/001-omen-pc-wsl-gpu-path/README.md` verifies that `/home/pmpg/rayme/.venv-cu121` imports `torch 2.5.1+cu121`, `triton 3.1.0`, `deepspeed 0.18.9`, and `flash-attn 2.8.3` on the RTX 3060.
- `.planning/spikes/002-f5-triton-trtllm-wsl-path/README.md` verifies the safe WSL Triton/TensorRT-LLM deployment path and already contains a short-utterance comparison against native Windows F5.
</context>

<must_haves>
- Persist a single matrix artifact that makes it impossible to confuse native-Windows measurements with WSL-backed measurements.
- F5 coverage must include both short-response and long-form / production-shaped comparison rows.
- XTTS coverage must distinguish baseline execution from any DeepSpeed-enabled path that is actually available; if DeepSpeed cannot be engaged or produces no measurable change, record that explicitly rather than silently collapsing the rows.
- Qwen coverage must distinguish eager from FlashAttention 2 when the WSL accelerator env makes both paths practical; if only one path is runnable, record the missing path and the reason.
- The resulting artifact must state which host account and runtime executed each row (`rayme-ssh` Windows, `rayme-pmpg` WSL Python, or `rayme-pmpg` WSL Triton).
</must_haves>

<tasks>
1. Build `probes/tts_runtime_matrix.py` and tests for its comparison/schema helpers. The harness may reuse existing `tts_ttfa.py`, `f5_production_chunking.py`, and spike scripts where appropriate rather than duplicating the engine logic.
2. Stage shared Phase 0 voice fixtures into every runtime under test, including the short reference clip and the longer F5 target text.
3. Measure F5 across the runtimes that matter to the decision:
   - native Windows baseline on `OMEN-PC`
   - WSL Python path on `OMEN-PC`
   - WSL Triton/TensorRT-LLM path on `OMEN-PC`
   - capture both short-ack and long-form / production-shaped rows when the path supports them
4. Measure XTTS across baseline and DeepSpeed-enabled execution in WSL. If XTTS does not expose a practical DeepSpeed toggle in the pinned stack, emit an explicit `not_available` or `no_effect_observed` row instead of omitting it.
5. Measure Qwen across eager and FlashAttention 2 in WSL using the same reference fixture and target text shape. Keep the eager baseline and FA2 row separate in the output JSON.
6. Write `results/tts_runtime_matrix.json` with per-row runtime metadata, TTFA, RTF, peak VRAM, source artifact pointers, and a concise recommendation block summarizing what changed materially and what did not.
</tasks>
