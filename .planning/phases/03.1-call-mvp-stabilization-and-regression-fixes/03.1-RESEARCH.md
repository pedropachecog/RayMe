### Locked Decisions

#### Regression Scope
- **D-01:** Phase 03.1 covers known defects plus nearby brittle edges, not a broad open-ended call rewrite.
- **D-02:** Brittle edges must be found evidence-first from Phase 3/8 evidence, logs, flaky tests, Android/manual acceptance issues, and then limited to directly connected call startup, teardown, reconnect, mute, interrupt, and writeback paths.
- **D-03:** Android/mobile blockers are the highest-priority defect class inside this stabilization phase.
- **D-04:** Targeted local refactors are allowed when they make the stabilization fix cleaner and less fragile, as long as they stay inside the affected call path.
- **D-05:** Small case-by-case implementation calls are left to planning/implementation, but they must respect the stabilization boundary and avoid new user-facing call features unless needed for recovery or correctness.

#### Acceptance Gate
- **D-06:** Completion requires local automated tests plus OMEN live desktop verification. Physical Android Chrome product-owner acceptance is required when changes touch mobile Chrome behavior, mic permission, call startup, audio playback, or layout.
- **D-07:** The local baseline is the focused call regression suite plus impacted tests: AI backend call/WebRTC tests, Web UI call/prompt tests, client call unit tests, relevant Playwright call specs, and any additional tests for touched files.
- **D-08:** OMEN verification must use canonical deployment through `scripts/deploy-omen.sh`, then run non-mocked live call evidence against `https://192.168.1.199:8443`.
- **D-09:** Phase 03.1 should produce a dedicated evidence document. Use machine-readable logs or JSON snippets where they materially support runtime-sensitive checks.

#### TTS Engine Coverage
- **D-10:** Affected call paths must prove both `voxcpm2` as the current preferred/default live-call engine and `f5` as the Phase 3 known-good fallback.
- **D-11:** Dual-engine proof should be behavioral smoke per engine. Add deeper timing, chunking, or cancel assertions only when the stabilization work touches those paths.
- **D-12:** If either `voxcpm2` or `f5` fails live call verification, Phase 03.1 is blocked unless the failure is clearly proven unrelated to this phase and documented as an OMEN setup/runtime follow-up.

#### Reconnect And Failure Behavior
- **D-13:** Mid-call media/WebRTC drops should try bounded recovery, then end truthfully and preserve the transcript if recovery fails.
- **D-14:** User-facing failure handling must prefer honest failed/ended states over optimistic continuation when RayMe cannot prove the call is still usable.
- **D-15:** Explicit regression coverage should include startup, reconnect/drop, mute, interrupt, end-call, and durable writeback.
- **D-16:** Reconnect/backfill work should stabilize the existing recovery/backfill behavior only. Do not redesign reconnect as a new visible feature in this phase.

#### Diagnostics And Evidence
- **D-17:** Keep useful live-call diagnostics, but gate or bound noisy debug plumbing so production behavior does not turn into unbounded log spam.
- **D-18:** Diagnostics may be broad when needed to debug live call failures, but secrets remain forbidden: no API keys, passwords, tokens, or other sensitive credentials in planning documents, commits, or evidence.
- **D-19:** Verbose/raw diagnostics should live in temporary or runtime artifacts. Committed Phase 03.1 evidence should contain sanitized summaries and useful excerpts only.
- **D-20:** Machine-readable evidence is required for runtime-sensitive checks only, such as engine smoke, mute frame drops, reconnect outcome, and call writeback counts. Do not generate machine artifacts for every acceptance item by default.

#### Scope Boundary
- **D-21:** Phase 4 call-feel capabilities remain out of scope: VAD barge-in, echo-loop mitigation, TTFA target optimization, advanced call-feel tuning, full visualizer polish, and new in-call features.
- **D-22:** If stabilization uncovers a real issue outside Phase 03.1, document and defer it unless it blocks current MVP correctness.
- **D-23:** User-facing UI changes should be limited to small recovery, status, or copy fixes needed for truthful stabilization. Do not redesign the call surface.
- **D-24:** Deferred Phase 4/backlog candidates should be recorded as deferred ideas, not silently converted into implementation tasks.

### Claude's Discretion

- Exact test command grouping beyond the required focused call baseline.

- Broad call UI redesign is out of scope.

Phase 03.1 should be planned as a narrow stabilization pass over the shipped Phase 3 call MVP, not as Phase 4 call-feel work. The highest-risk areas are the cross-tier call lifecycle boundaries: browser mic/WebRTC startup, bounded reconnect/backfill, server-side mute and interrupt, truthful end/fail handling, durable transcript/writeback, and live verification for both `voxcpm2` and `f5`. The planner should extend and harden these surfaces rather than introducing new call protocols or user-visible features. ### Pattern 1: Evidence-First Stabilization

**What:** start by reading Phase 3/8 evidence, live logs, flaky tests, Android acceptance notes, and the existing regression suites before selecting fixes. [VERIFIED: 03.1-CONTEXT.md]

**When to use:** use for every candidate defect, especially reconnect/drop, Android mic/audio, or engine behavior. [VERIFIED: 03.1-CONTEXT.md]

**Implementation guidance:** add or extend tests first for the observed defect, then make the smallest code change inside the affected path. [VERIFIED: 03.1-CONTEXT.md; VERIFIED: .planning/LEARNINGS.md]

### Pattern 2: Browser Owns Media Setup, Server Owns Durable State

**What:** browser code requests mic permission, unlocks audio, creates `RTCPeerConnection`, watches `connectionState` and `iceConnectionState`, and captures local PCM for reconnect backfill. [VERIFIED: web-ui/client/src/routes/call/[threadId]/+page.svelte; CITED: https://developer.mozilla.org/en-US/docs/Web/API/MediaDevices/getUserMedia; CITED: https://developer.mozilla.org/en-US/docs/Web/API/RTCPeerConnection/connectionState]

**When to use:** plan browser tasks for permission, Android, AudioContext, WebRTC state, local UI status, and local PCM backfill defects. [VERIFIED: web-ui/client/src/routes/call/[threadId]/+page.svelte]

**Do not move:** durable transcript/writeback logic should stay in `CallService` and Web UI server routes. [VERIFIED: web-ui/server/app/domain/call_service.py]

### Pattern 3: Same-Origin Call Facade

**What:** browser controls call the Web UI server under `/api/calls`; the server enforces same-origin and forwards validated requests to the AI backend. [VERIFIED: web-ui/server/app/api/calls.py]

**When to use:** plan server tasks for `start`, `offer`, `mute`, `interrupt`, `turns`, `reconnect-audio`, `events/recover`, `end`, and sanitized diagnostics. [VERIFIED: web-ui/server/app/api/calls.py]

**Why:** this keeps browser-facing errors and durable state under RayMe control while the AI backend remains a transient media orchestrator. [VERIFIED: .planning/STATE.md; VERIFIED: web-ui/server/app/api/calls.py]

### Pattern 4: Bounded Reconnect With Truthful Terminal State

**What:** browser schedules bounded reconnect attempts, sends backfill, recovers missed backend events, and ends truthfully when recovery cannot prove usability. [VERIFIED: web-ui/client/src/routes/call/[threadId]/+page.svelte]

**When to use:** plan reconnect fixes around existing constants and flow, not a new visible reconnect feature. [VERIFIED: 03.1-CONTEXT.md; VERIFIED: web-ui/client/src/routes/call/[threadId]/+page.svelte]

**External contract:** MDN describes `iceConnectionState` `disconnected` as less strict and potentially transient, while `failed` means compatible ICE candidates could not be found for all components. [CITED: https://developer.mozilla.org/en-US/docs/Web/API/RTCPeerConnection/iceConnectionState]

### Pattern 5: Dual-Engine Behavioral Smoke

**What:** prove affected call paths with `voxcpm2` and `f5`; use deeper chunk/timing assertions only when stabilization touches those paths. [VERIFIED: 03.1-CONTEXT.md]

**When to use:** every local or live test plan that touches TTS playback, voice metadata, `/webrtc/speak`, interrupt, reconnect, or end-of-call writeback. [VERIFIED: 03.1-CONTEXT.md; VERIFIED: ai-backend/app/call/session.py]

**Important gap:** `web-ui/client/tests/e2e/live-call.spec.ts` currently configures `tts_default_engine: 'f5'` and creates an `f5` voice fixture, so it is not sufficient by itself for Phase 03.1 dual-engine proof. [VERIFIED: web-ui/client/tests/e2e/live-call.spec.ts]

### Anti-Patterns to Avoid

- **Broad call rewrite:** it contradicts D-01 and risks regressing Phase 3 acceptance. [VERIFIED: 03.1-CONTEXT.md]
- **Phase 4 feature creep:** VAD barge-in, echo-loop mitigation, TTFA target tuning, advanced visualizer polish, and new call features are out of scope. [VERIFIED: 03.1-CONTEXT.md]
- **Direct browser-to-AI-backend controls:** existing policy uses the Web UI same-origin facade for call controls. [VERIFIED: web-ui/server/app/api/calls.py; VERIFIED: .planning/STATE.md]
- **Unbounded diagnostics:** `_debug/event` exists for browser diagnostic mirroring, but committed evidence must be sanitized and debug plumbing must not become unbounded production spam. [VERIFIED: web-ui/server/app/api/calls.py; VERIFIED: 03.1-CONTEXT.md]
- **Ad-hoc OMEN workarounds:** AGENTS.md forbids alternate deployment scripts and manual scheduled-task manipulation. [VERIFIED: ./AGENTS.md]

## Common Pitfalls

### Pitfall 1: Treating Phase 3 F5 Live Acceptance as Phase 8 VoxCPM2 Proof

**What goes wrong:** a plan runs the existing live spec and claims dual-engine coverage even though the spec configures `f5`. [VERIFIED: web-ui/client/tests/e2e/live-call.spec.ts]

**Why it happens:** Phase 3 live acceptance predates Phase 8 promotion of `voxcpm2`. [VERIFIED: .planning/phases/03-first-working-call-mvp/OMEN-PC-LIVE-EVIDENCE.md; VERIFIED: .planning/STATE.md]

**How to avoid:** add parameterized or separate behavioral smoke for `voxcpm2` and `f5`, and write evidence naming both engines. [VERIFIED: 03.1-CONTEXT.md]

**Warning signs:** evidence contains only `default_engine: 'f5'`, no `voxcpm2` voice fixture, or no nested `ai_audio_started` / playback proof for VoxCPM2. [VERIFIED: web-ui/client/tests/e2e/live-call.spec.ts; VERIFIED: .planning/phases/08-wire-voxcpm2-streaming-chunks-into-live-rayme-call-playback/08-VERIFICATION.md]

### Pitfall 2: Updating Call Defaults Without Updating Deploy Health Guards

**What goes wrong:** a plan expects `voxcpm2` as the live resident engine, but `scripts/deploy-omen.sh` still fails health if AI or Web UI reports resident engine other than `f5`. [VERIFIED: scripts/deploy-omen.sh]

**Why it happens:** Phase 8 promoted `voxcpm2` at the call-behavior decision layer, while the deployment script's baseline health guard still encodes F5 residency. [VERIFIED: .planning/STATE.md; VERIFIED: scripts/deploy-omen.sh]

**How to avoid:** either prove `voxcpm2` through per-voice call engine selection while keeping deploy residency guard unchanged, or explicitly plan a deploy-script guard update with tests/evidence. [VERIFIED: web-ui/server/app/domain/call_service.py; VERIFIED: scripts/deploy-omen.sh]

**Warning signs:** live deploy fails with "resident F5" health errors after a default-engine change. [VERIFIED: scripts/deploy-omen.sh]

### Pitfall 3: Reapplying Known-Bad Reconnect Patch Stack

**What goes wrong:** a small-looking reconnect patch reintroduces the post-snapshot phone-call regression. [VERIFIED: .planning/LEARNINGS.md]

**Why it happens:** reconnect logic crosses browser WebRTC state, local PCM buffers, backend media state, and durable writeback. [VERIFIED: web-ui/client/src/routes/call/[threadId]/+page.svelte; VERIFIED: ai-backend/app/call/session.py; VERIFIED: web-ui/server/app/api/calls.py]

**How to avoid:** start from failing evidence, add targeted tests, and keep changes inside the existing bounded recovery/backfill path. [VERIFIED: 03.1-CONTEXT.md; VERIFIED: .planning/LEARNINGS.md]

**Warning signs:** a plan proposes a new reconnect architecture, applies commits `6f63de0`/`a0d5d17`, or skips live phone evidence for mobile-sensitive work. [VERIFIED: .planning/LEARNINGS.md; VERIFIED: 03.1-CONTEXT.md]

### Pitfall 4: Misclassifying WebRTC `disconnected` as Always Terminal

**What goes wrong:** transient ICE loss ends the call too aggressively or skips bounded reconnect/backfill. [CITED: https://developer.mozilla.org/en-US/docs/Web/API/RTCPeerConnection/iceConnectionState]

**Why it happens:** MDN distinguishes `disconnected` as less strict and potentially transient, while `failed` is a stronger failure signal. [CITED: https://developer.mozilla.org/en-US/docs/Web/API/RTCPeerConnection/iceConnectionState]

**How to avoid:** preserve the existing grace/retry behavior for `disconnected`, but end truthfully after bounded attempts fail. [VERIFIED: web-ui/client/src/routes/call/[threadId]/+page.svelte; VERIFIED: 03.1-CONTEXT.md]

**Warning signs:** tests pass for `failed` but lack `disconnected` grace and recovered-connection assertions. [VERIFIED: web-ui/client/tests/e2e/call-start.spec.ts]

### Pitfall 5: Losing Transcript Rows During Failure Cleanup

**What goes wrong:** the call fails or ends before missed `user_final` events are recovered and durable rows are written. [VERIFIED: web-ui/client/src/routes/call/[threadId]/+page.svelte; VERIFIED: web-ui/server/app/api/calls.py]

**Why it happens:** data-channel delivery can fail during teardown, and `CallSession` queues only selected recoverable event types. [VERIFIED: ai-backend/app/call/session.py]

**How to avoid:** keep client hangup/terminal cleanup order: flush backfill, recover missed events, then end call; server end must preserve `call_end` writeback when backend control failure is non-misconfiguration. [VERIFIED: web-ui/client/src/routes/call/[threadId]/+page.svelte; VERIFIED: web-ui/server/app/api/calls.py]

**Warning signs:** failing tests around `events/recover`, `call_end`, or return-to-thread row counts. [VERIFIED: web-ui/client/tests/e2e/call-start.spec.ts; VERIFIED: web-ui/client/tests/e2e/call-summary.spec.ts]

### Pitfall 6: Treating curl Health as Browser Acceptance

**What goes wrong:** deployment is called ready after `/health` checks but before a rendered browser call path is exercised. [VERIFIED: .planning/OPERATING-NOTES.md]

**Why it happens:** health checks prove service reachability, not WebRTC, mic, audio, UI, or transcript behavior. [VERIFIED: web-ui/client/tests/e2e/live-call.spec.ts]

**How to avoid:** run browser Playwright evidence after canonical deployment, then request Android product-owner acceptance only when required by touched scope. [VERIFIED: 03.1-CONTEXT.md; VERIFIED: .planning/OPERATING-NOTES.md]

**Warning signs:** evidence has only `curl` output and no Playwright trace/screenshots/log summary. [VERIFIED: .planning/OPERATING-NOTES.md]
