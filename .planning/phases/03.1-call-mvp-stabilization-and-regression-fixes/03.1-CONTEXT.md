# Phase 03.1: Call MVP Stabilization and Regression Fixes - Context

**Gathered:** 2026-05-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 03.1 stabilizes the already-shipped Phase 3 call MVP before Phase 4 call-feel work begins. It fixes known call defects plus nearby brittle edges, with priority on Android/mobile blockers and the existing browser/WebRTC call lifecycle: startup, negotiation, reconnect/drop/backfill, mute, interrupt, end-call, TTS playback smoke, and durable thread writeback.

This phase does not add Phase 4 call-feel capabilities. VAD barge-in, echo-loop mitigation, TTFA optimization targets, full visualizer polish, advanced caption behavior, and new in-call features stay out of scope unless a narrow fix is required to preserve current MVP correctness.

</domain>

<decisions>
## Implementation Decisions

### Regression Scope
- **D-01:** Phase 03.1 covers known defects plus nearby brittle edges, not a broad open-ended call rewrite.
- **D-02:** Brittle edges must be found evidence-first from Phase 3/8 evidence, logs, flaky tests, Android/manual acceptance issues, and then limited to directly connected call startup, teardown, reconnect, mute, interrupt, and writeback paths.
- **D-03:** Android/mobile blockers are the highest-priority defect class inside this stabilization phase.
- **D-04:** Targeted local refactors are allowed when they make the stabilization fix cleaner and less fragile, as long as they stay inside the affected call path.
- **D-05:** Small case-by-case implementation calls are left to planning/implementation, but they must respect the stabilization boundary and avoid new user-facing call features unless needed for recovery or correctness.

### Acceptance Gate
- **D-06:** Completion requires local automated tests plus OMEN live desktop verification. Physical Android Chrome product-owner acceptance is required when changes touch mobile Chrome behavior, mic permission, call startup, audio playback, or layout.
- **D-07:** The local baseline is the focused call regression suite plus impacted tests: AI backend call/WebRTC tests, Web UI call/prompt tests, client call unit tests, relevant Playwright call specs, and any additional tests for touched files.
- **D-08:** OMEN verification must use canonical deployment through `scripts/deploy-omen.sh`, then run non-mocked live call evidence against `https://192.168.1.199:8443`.
- **D-09:** Phase 03.1 should produce a dedicated evidence document. Use machine-readable logs or JSON snippets where they materially support runtime-sensitive checks.

### TTS Engine Coverage
- **D-10:** Affected call paths must prove both `voxcpm2` as the current preferred/default live-call engine and `f5` as the Phase 3 known-good fallback.
- **D-11:** Dual-engine proof should be behavioral smoke per engine. Add deeper timing, chunking, or cancel assertions only when the stabilization work touches those paths.
- **D-12:** If either `voxcpm2` or `f5` fails live call verification, Phase 03.1 is blocked unless the failure is clearly proven unrelated to this phase and documented as an OMEN setup/runtime follow-up.

### Reconnect And Failure Behavior
- **D-13:** Mid-call media/WebRTC drops should try bounded recovery, then end truthfully and preserve the transcript if recovery fails.
- **D-14:** User-facing failure handling must prefer honest failed/ended states over optimistic continuation when RayMe cannot prove the call is still usable.
- **D-15:** Explicit regression coverage should include startup, reconnect/drop, mute, interrupt, end-call, and durable writeback.
- **D-16:** Reconnect/backfill work should stabilize the existing recovery/backfill behavior only. Do not redesign reconnect as a new visible feature in this phase.

### Diagnostics And Evidence
- **D-17:** Keep useful live-call diagnostics, but gate or bound noisy debug plumbing so production behavior does not turn into unbounded log spam.
- **D-18:** Diagnostics may be broad when needed to debug live call failures, but secrets remain forbidden: no API keys, passwords, tokens, or other sensitive credentials in planning documents, commits, or evidence.
- **D-19:** Verbose/raw diagnostics should live in temporary or runtime artifacts. Committed Phase 03.1 evidence should contain sanitized summaries and useful excerpts only.
- **D-20:** Machine-readable evidence is required for runtime-sensitive checks only, such as engine smoke, mute frame drops, reconnect outcome, and call writeback counts. Do not generate machine artifacts for every acceptance item by default.

### Scope Boundary
- **D-21:** Phase 4 call-feel capabilities remain out of scope: VAD barge-in, echo-loop mitigation, TTFA target optimization, advanced call-feel tuning, full visualizer polish, and new in-call features.
- **D-22:** If stabilization uncovers a real issue outside Phase 03.1, document and defer it unless it blocks current MVP correctness.
- **D-23:** User-facing UI changes should be limited to small recovery, status, or copy fixes needed for truthful stabilization. Do not redesign the call surface.
- **D-24:** Deferred Phase 4/backlog candidates should be recorded as deferred ideas, not silently converted into implementation tasks.

### the agent's Discretion
- Exact plan split across backend, server, client, and evidence work.
- Exact local refactor shape for brittle call lifecycle code, as long as it remains scoped to the affected path.
- Exact test command grouping beyond the required focused call baseline.
- Exact diagnostic gating mechanism and artifact names, provided committed evidence is sanitized.
- Exact machine-readable fields for runtime-sensitive evidence, provided they prove the selected acceptance gates.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `CallSession` already owns listening/understanding/thinking/speaking/ended state, inbound VAD/STT finalization, muting, interrupts, reconnect grace, reconnect audio backfill, undelivered event recovery, and TTS playback.
- `web-ui/server/app/api/calls.py` already exposes same-origin lifecycle routes plus `/turns`, `/reconnect-audio`, `/events/recover`, and `/_debug/event` paths that can be stabilized or gated instead of inventing new surfaces.
- `CallService` already owns durable `call_start`, `user_speech`, `ai_speech`, and `call_end` writes, active `call_` to `rtc_` mapping, voice preflight, and VoxCPM2 call metadata forwarding.
- The client call route already records local mic PCM chunks, attempts bounded media reconnects, sends reconnect backfill, recovers missed events, and forwards browser diagnostics.
- The Phase 3 and Phase 8 test suites already cover much of the desired baseline and should be extended rather than bypassed.

### Established Patterns
- Durable app state belongs to the Web UI server; transient media/runtime work belongs to the AI backend.
- Browser call controls use same-origin Web UI routes, not direct provider/runtime routes.
- Public errors and evidence must be sanitized; raw traces, secrets, full local paths, and sensitive runtime details do not belong in committed docs.
- Live OMEN work must deploy through `scripts/deploy-omen.sh`; do not create alternate OMEN launchers, scripts, scheduled-task edits, or hidden process workarounds.
- Manual Android testing is product-owner acceptance after agent-run local and OMEN verification, not first-line QA.

### Integration Points
- Stabilization likely touches the call route (`+page.svelte`), Web UI call facade (`calls.py`), `CallService`, AI backend `CallSession`, and WebRTC routes.
- Test work should start from existing call unit/API/Playwright specs and the live-call spec, then add focused regressions for the touched path.
- Evidence work should produce a Phase 03.1-specific evidence file and only add JSON/log snippets for runtime-sensitive proof.

</code_context>

- User later directed accepting the remaining recommended values, so Diagnostics/Evidence and Scope Boundary include recommended defaults after the explicitly selected diagnostics data-boundary exception.
