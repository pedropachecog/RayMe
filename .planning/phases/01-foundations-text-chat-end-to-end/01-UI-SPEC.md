# Phase 01 - UI Design Contract

> Visual and interaction contract for Phase 1 frontend work. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Scope And Sources

Phase 1 ships the actual app shell, not a landing page. The primary user path is: Home -> Character Gallery import/create -> Character Editor review/save -> Chat -> streaming LLM reply -> persisted unified messages -> reload and continue.

## Design System

| Property | Value |
|----------|-------|
| Tool | none - manual Stitch-derived design system |
| Preset | not applicable |
| Component library | local Svelte components; no shadcn/Radix registry in Phase 1 |
| Icon library | `lucide-svelte`; map Stitch Material Symbols to nearest lucide icons |
| Font | Manrope for display/heading, Inter for body/label |

Design-system contract:

- Use the `docs/stitch/DESIGN.md` tokens as canonical when the static HTML export palette differs. Use the HTML exports for layout feel and component proportions, not as an override for the design-system colors.
- Preserve the True Dark, tonal-layered interface: no bright app chrome, no pure white text, no account/billing/logout affordances, and no marketing hero.
- Apply the no-line rule for layout sectioning. Use background shifts and spacing for structure. Only use ghost outlines for focus, form affordance, drag/drop affordance, and destructive confirmation.
- Glass surfaces are allowed for floating menus, modals, composer, and sticky headers: `surface-container-highest` at 60% opacity with `backdrop-filter: blur(20px)`.
- Do not nest UI cards inside other cards. Use full-width page bands, split panes, or unframed layouts for sections; repeated records may be cards.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, dense message action spacing |
| sm | 8px | Compact element spacing, message stack gaps, grid gutters on mobile |
| md | 16px | Default element spacing, input padding, card internal padding |
| lg | 24px | Section padding, editor form group spacing, desktop grid gutters |
| xl | 32px | Page gutters, shell content gap, chat rail-to-thread gap |
| 2xl | 48px | Major vertical separation, empty-state breathing room |
| 3xl | 64px | Desktop top/bottom workspace padding only |

Exceptions: minimum interactive target is 44px for icon buttons, nav items, card action menus, message overflow controls, and mobile composer actions. This is the only 8-point-scale exception and remains a multiple of 4.

Layout dimensions:

- Desktop side navigation: 80px collapsed icon rail or 240px expanded rail; pick one implementation and keep it stable across Phase 1 screens.
- Mobile bottom navigation: 64px high, fixed, with Home, Gallery, and Settings only. Chat and Editor are contextual screens with a back affordance.
- Main content max width: 1440px on wide desktop; chat readable column max width 920px unless a thread rail is present.
- Repeated character cards: stable aspect ratio of 4:5 for portrait media, with content area below. Cards must not resize on hover or action-menu open.
- Composer: 56px minimum collapsed height, grows up to 168px before internal scrolling.

## Typography

Use exactly these four type sizes for Phase 1 UI text.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 600 | 1.3 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.15 |

Typography rules:

- Use only weights 400 and 600.
- Letter spacing is 0.
- Use Display only for screen-level greetings/titles, never inside cards, badges, buttons, or compact panels.
- Chat message text uses Body. Character prose fields render at Body with 1.5 line height.
- Buttons and nav labels use Label.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#060e20` | App background, chat canvas, page base |
| Secondary (30%) | `#091328`, `#141f38`, `#192540` | Sidebar, panes, cards, editor panels, menus, composer |
| Accent (10%) | `#b6a0ff`; pulse gradient `linear-gradient(135deg, #b6a0ff 0%, #70aaff 100%)` | Specific elements listed below |
| Destructive | `#ff716c` | Destructive actions only |

Accent reserved for: primary CTA, active nav item, selected alternate greeting, selected swipe index, focused composer glow, streaming AI caret, import dropzone active state, healthy/secure status emphasis, and Save button.

Additional color rules:

- Communication/confirmation states may use `#00e3fd` only as a small chip dot, icon, or glow. Pair it with text such as `Connected`, `Secure`, or `Streaming`; color must not carry state alone.
- Text uses `#dee5ff` for primary text and `#9eaad5` for secondary text. Do not use `#ffffff`.
- Ghost outlines use `#40485d` at 15% opacity for passive structure and at stronger opacity only for keyboard focus.
- Destructive controls are text or outline by default. Fill with destructive color only inside confirmation dialogs.
- Sanitized untrusted character-card content must use the same text colors as app prose and must not inherit arbitrary card-provided color styles.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Start Chat |
| Empty state heading | No characters yet |
| Empty state body | Import a SillyTavern card or create a character to start your first chat. |
| Error state | RayMe cannot reach the LLM endpoint. Check Settings, run Test Connection, and try again. |
| Destructive confirmation | Delete character: Delete this character? Existing chats stay in history, but the character leaves the gallery. |
| Destructive confirmation | Delete thread: Delete this thread? This removes the conversation history. |
| Destructive confirmation | Truncate stale turns: Remove stale turns after this edit? The selected message stays and later turns are removed from this branch. |
| Destructive confirmation | Discard edits: Discard unsaved changes? Your last saved character version will remain. |

Copy rules:

- Use direct operational labels: `Import Character`, `Create Character`, `Save Character`, `Export JSON`, `Test Connection`, `Send`, `Regenerate`, `Edit`, `Continue`.
- Avoid future-facing labels for Phase 2+ features. Do not show disabled Voice Lab, Call, billing, account, logout, or voice-model controls in Phase 1.
- Error copy must state the failing dependency and the next action. Toasts may be short, but inline errors on forms must be specific.

## Screen Contracts

### App Shell

- Desktop uses a persistent left navigation rail and one main workspace. The rail contains Home, Gallery, and Settings for Phase 1. Editor and Chat are reached contextually and should not appear as top-level fake destinations.
- Mobile uses a fixed bottom navigation with Home, Gallery, and Settings. Editor and Chat use a top back affordance.
- The first screen after load is Home. Never show a marketing landing page.
- Shell status affordances must expose secure context and endpoint health without dominating the page. Use compact chips, not banners, when everything is healthy.
- If `window.isSecureContext` is false or `navigator.mediaDevices` is absent, show a persistent setup warning in Home and Settings. Text chat may still work, but the warning must make mobile readiness visible.

### Home

- Home is threads-first. The recent thread list is the largest information surface.
- Required thread row content: character portrait, character name, thread title or fallback character name, last-message snippet, relative timestamp, and contextual menu for rename/delete.
- Primary action is `Start Chat`. If no characters exist, route the primary action to Gallery with the empty state focused on import/create.
- Secondary actions: `Import Character`, `Create Character`, and `Settings`.
- Do not include Stitch mock usage stats, quick-start personas, notifications, account controls, or logout.
- Empty thread state: show `No conversations yet` as the local heading and route to `Start Chat`; keep the global empty character copy for the no-character state.

### Character Gallery

- Gallery is the import entry point. Primary visible actions are `Import Character` and `Create Character`.
- Grid cards show portrait, character name, short sanitized description snippet, tag count or first two tags when present, and an overflow menu.
- Card actions: `Start Chat`, `Edit`, `Export JSON`, `Delete`.
- No search, tag filters, or voice assignment badges in Phase 1.
- Import interaction accepts JSON and PNG. While parsing, show a tonal loading row or modal. On success, open Character Editor in review mode before final save.
- Import warnings use inline chips: `Lorebook present - not used in v1`, `Fields preserved`, or `Unsupported fields ignored`. These warnings must not block save unless parsing is unsafe.
- Malicious HTML from a card renders as escaped/sanitized text. Never render raw card HTML.

### Character Editor

- Desktop editor uses a two-column layout: identity/media/meta on the left, prompts/greetings/advanced text fields on the right. Mobile stacks the same sections in this order.
- Header actions: `Discard Edits` and `Save Character`. Save is accent; Discard Edits is secondary.
- Required fields visible in Phase 1: name, description, personality, scenario, first message, example messages, system prompt, creator notes, character notes, tags, alternate greetings, post-history instructions, creator, character version, and portrait upload/preview.
- Use text areas for long character-card fields. Every long field has stable height and internal scroll after its max height; typing must not push header or footer controls off-screen.
- Alternate greetings use a list editor with add, edit, remove, and reorder controls. Use compact icon buttons with accessible labels.
- Lorebook/world-info is read-only in Phase 1. Show a compact indicator when present: `Lorebook present - not used in v1`.
- Do not show voice selection, voice preview, templates, optimizer, or SillyTavern compatibility toggles as active controls in Phase 1.
- Portrait upload uses a large dropzone with preview, replace, and remove actions. Failed image import errors must name the accepted formats.

### Chat

- Chat is a spacious modern messenger, not a SillyTavern clone. Use the RayMe tonal surfaces while retaining enough density for long threads.
- Header shows character portrait, character name, current thread title, and a menu for rename/delete. Do not show a nonfunctional call button in Phase 1.
- New chat creation inserts the character `first_mes` as the opening AI turn. If alternate greetings exist, show an alternate-greeting control attached to the opening AI message until the first user message is sent.
- Message alignment: user text aligns right or end; AI text aligns left or start with character portrait context. Both use readable bubble widths, not full-row cards.
- Streaming AI output appears in a single AI message bubble with token-by-token text and an accent streaming caret. The final persisted message replaces the streaming buffer without layout jump.
- Composer supports multiline typing. `Send` is an icon+label or icon button with accessible label. Enter sends and Shift+Enter inserts a newline.
- Message actions are contextual: AI turns expose Regenerate, Swipe, Edit, and Continue; user turns expose Edit. Desktop uses hover/focus menu; mobile uses tap overflow.
- Regenerate replaces the currently selected AI response for the turn. It must not append a second canonical answer.
- Swipes use a compact stepper: previous, current index, next, and generate alternate. Only the selected alternate is visually canonical.
- Editing any prior turn marks downstream turns stale. Stale turns get a subtle left rail and `Stale` chip. Continuing after an edit opens the truncate-or-keep choice.
- Continue uses composer text when present as the instruction/prefix to extend the previous AI turn.
- Jump-to-latest appears when the user scrolls away from the bottom. It is a floating glass control above the composer and must not cover the input text.
- Virtualization activates around 500 messages. Virtualized rows must preserve stable message heights and scroll position across streaming updates.

### Settings

- Settings in Phase 1 contains real endpoint settings only: Web UI status, AI backend URL/status, LLM URL, API key, model, LLM status, HTTPS secure-context status, and media-device availability status.
- Each endpoint section has a `Test Connection` action and a text status: `Connected`, `Unreachable`, `Unauthorized`, or `Not configured`.
- API key input is masked by default with a reveal icon button. The key is never echoed in status copy.
- Use Settings to surface mkcert/LAN trust readiness through checks, not through long setup prose. Link or route to documentation only if implementation includes docs.
- Do not show account, subscription, billing, wake word, screen awareness, VAD sensitivity, audio device defaults, save-audio toggles, clear-data danger zone, or PWA install controls in Phase 1.

## Interaction And State Contracts

### Loading

- Use skeletons built from tonal surfaces for Home rows, Gallery cards, Editor sections, Settings panels, and chat history.
- Use compact inline progress for import parsing and connection tests.
- Avoid full-screen spinners unless the app is bootstrapping configuration and cannot render the shell.

### Empty

- Empty states include one heading, one sentence body, and one primary action.
- Empty Home with no threads but existing characters points to `Start Chat`.
- Empty Gallery uses the global copy in the Copywriting Contract.
- Empty Chat should not occur after creation because the opening AI greeting is inserted immediately.

### Error

- LLM streaming failure keeps the user message and replaces the AI streaming bubble with an inline error plus retry/regenerate action.
- Import failure distinguishes unsupported file, malformed JSON, unreadable PNG metadata, and unsafe content.
- Connection-test failure identifies which endpoint failed. Do not collapse all failures into a generic network error.
- Insecure-context warning persists until the browser reports a secure context.

### Focus And Keyboard

- Every icon-only button has an accessible label and visible focus.
- Keyboard focus uses a 2px accent or ghost outline and does not rely on hover.
- Dialogs trap focus and close on Escape unless a destructive operation is currently submitting.
- Menus are reachable by keyboard and touch. Hover-only controls must reveal on focus.

### Motion

- Use motion sparingly: streaming caret pulse, status-chip transition, skeleton shimmer, and menu/dialog fade.
- Respect `prefers-reduced-motion` by disabling shimmer, pulse, and animated gradient movement.
- No decorative orbs or bokeh blobs. Background depth comes from tonal surfaces and glass layering.

## Responsive Contract

- Desktop breakpoint: navigation rail + multi-column Home/Gallery/Editor.
- Tablet: rail may collapse; content remains two-column only if each column keeps at least 360px.
- Mobile: single-column screens, bottom nav, sticky composer, and 44px minimum targets.
- Text must never overflow buttons, chips, menus, or cards. Long character names and thread titles truncate with accessible full text where needed.
- Chat composer and jump-to-latest must not overlap mobile bottom navigation.
- Editor Save Character/Discard Edits actions remain reachable on mobile through a sticky bottom action bar or sticky header, but never both.

---

## Accessibility And Safety

- Color is never the only state indicator. Pair every health/error/stale/selected state with text, icon, or shape.
- Sanitized Markdown is the only rendering path for imported character text. Raw HTML from cards is not allowed.
- Confirm destructive actions before execution. The confirmation action button uses the destructive color and repeats the destructive verb.
- Preserve chat history on character delete per Phase 1 context. UI copy must not imply delete will remove historical threads.
- API keys are treated as sensitive UI content: masked input, no toast echo, no logging display.
