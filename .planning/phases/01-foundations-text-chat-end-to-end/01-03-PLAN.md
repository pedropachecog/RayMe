<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LAN browser -> AI backend health | Unauthenticated clients can probe the health-only stub. |
| web-ui/server -> external LLM | Sensitive API key and endpoint config leave RayMe only from server-side code. |
| LAN network -> web host | Direct IP HTTPS exposes the app on a configured interface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-03-BIND | I/T | `config.example.env`, HTTPS doc | mitigate | Document explicit `RAYME_WEB_BIND_HOST=192.168.1.199` and warn against `0.0.0.0`. |
| T-01-03-CORS | S/T | HTTPS doc | mitigate | Document `RAYME_ALLOWED_ORIGINS` and no wildcard CORS before web implementation. |
| T-01-03-KEY | I | `llm/README.md`, env example | mitigate | Keep API key in server env/settings only; use a blank example value and Settings probe contract. |
| T-01-03-SCOPE | T | `ai-backend/app/main.py`, `llm/README.md` | mitigate | Health-only AI backend and docs-only LLM prevent fake local model services from contaminating topology. |
| T-01-03-TLS | S/I | `docs/phase1-https-lan.md` | mitigate | Reuse mkcert trusted direct-IP workflow and secure-context checks from Phase 0. |
| T-01-03-AIHTTPS | S/I | `ai-backend/scripts/run_https.py`, `docs/phase1-https-lan.md` | mitigate | AI backend runner rejects wildcard binds, requires explicit cert/key, defaults/documents port 9443, and documents `https://192.168.1.199:9443/health` verification. |
</threat_model>
