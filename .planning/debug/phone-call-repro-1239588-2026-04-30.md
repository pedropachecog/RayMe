## First Loss Point

The loss is confirmed before STT and before Web UI persistence. The first
observable loss boundary is the media reconnect / VAD boundary: after the
replacement WebRTC track is established, the backend receive loop gets frames
but VAD sees no voiced speech before reconnect grace expires and finalizes the
partial turn.

Current logs do not distinguish whether the missing speech is already absent at
browser microphone capture, being sent as silent RTP over the new WebRTC track,
or present in RTP but rejected by VAD. The next experiment should add explicit
browser local-mic RMS logging around media reconnect and backend post-reconnect
frame RMS logging, then repeat the same passage.

## Current Finding

The post-`1239588` fix closed the previous failed-state grace-ordering bug:
`vad.reconnect_grace.pending/start` now appear in live logs. The remaining
failure is that no VAD-positive speech resumes after reconnect before grace
expires. A low-risk functional fix is not confirmed from the current evidence;
instrumentation is needed to separate browser capture loss from backend
VAD/received-audio loss.
